# RL Environment & STEVAL-STWINBX1 Project Guidelines

## 1. Workspace Architecture & Tree Versioning (CRITICAL)

The project uses a **Tree Architecture** to support generic, reusable environments:

```
Generic_Base/                    # Hardware-agnostic Phase 4 code (v1.0.0)
                                 # NEVER edit directly

Environments/                    # Completed environment branches
  └── Accelerometer_Env/        # First node: ISM330DHCX accelerometer RL env
      ├── Core/Src/main.c        # Phase 5/6 working code
      ├── RL_Brain/agent.py      # Python threshold policy agent
      └── Dell_2_Steval.ioc     # CubeMX configuration

Active_Trial/                    # Current working directory (cloned from a base)
SUMMARY.md                       # Changelog of successful environment implementations
TRIAL_LOG.md                     # Detailed development trial log
LESSONS_LEARNED.md               # Permanent rules learned from failures
```

### Base Selection Rule
**When starting a new trial, the user will specify which base to clone into `Active_Trial/`**, either:
- `Generic_Base/` - Hardware-agnostic starting point for new environments
- A specific node from `Environments/` (e.g., `Environments/Accelerometer_Env/`) - To extend an existing environment

**YOU ARE STRICTLY FORBIDDEN FROM EDITING FILES IN `Generic_Base/` OR ANY `Environments/` NODE DIRECTLY.**

## 2. Code Modification Boundaries (ABSOLUTE RULE)
When editing files inside `Active_Trial/`, **YOU ARE FORBIDDEN FROM MODIFYING ANY CODE OUTSIDE OF `/* USER CODE BEGIN */` AND `/* USER CODE END */` BLOCKS.**

Only modify code within these protected regions:
- `/* USER CODE BEGIN PV */` ... `/* USER CODE END PV */` (Private Variables)
- `/* USER CODE BEGIN 0 */` ... `/* USER CODE END 0 */`
- `/* USER CODE BEGIN 1 */` ... `/* USER CODE END 1 */`
- `/* USER CODE BEGIN 2 */` ... `/* USER CODE END 2 */` (Setup after initialization)
- `/* USER CODE BEGIN 3 */` ... `/* USER CODE END 3 */` (Main while loop)
- `/* USER CODE BEGIN 4 */` ... `/* USER CODE END 4 */`

All HAL initialization, peripheral setup, and system configuration generated by CubeMX must remain untouched.

## 3. The RL Development Loop
Follow this strict workflow for every task:
1. **Setup**: Delete the existing `Active_Trial/` directory (if it exists). The user specifies the base (Generic_Base or a specific Environments node). Copy the chosen base into `Active_Trial/`.
2. **Edit**: Make code changes strictly within `Active_Trial/` using the USER CODE boundaries.
3. **Compile**: Run `make -C Active_Trial/Debug all`. (Never run `make clean` followed by `make` unless requested).
4. **Fix**: If build fails, analyze output, correct code within boundaries, and recompile.
5. **Test**: Flash the board and monitor the serial output at `/dev/ttyACM0`. Wait for user confirmation.
6. **Promote**: When the user says "Success, promote it", create a new node in `Environments/<EnvName>/` with the completed work.

## 4. Hardware Constraints & Timing
- **Board**: STEVAL-STWINBX1 (STM32U585AI microcontroller)
- **NEVER use `HAL_Delay()`**. Use non-blocking `HAL_GetTick()` delta comparisons:
  ```c
  if ((HAL_GetTick() - last_tick) >= desired_interval_ms) {
      last_tick = HAL_GetTick();
      // action
  }
  ```

## 5. Versioning, Logging & Memory (CRITICAL)

### Semantic Versioning
- **Generic_Base** is always **v1.0.0**
- Each **Environment node** maintains its own versioning (e.g., Accelerometer_Env v1.0.1, v1.0.2...)
- **Active_Trial** increments the patch version for every new compilation (e.g., v1.0.1, v1.0.2, v1.0.3...)

### Mandatory Logging
Every time I compile and test code, **regardless of success or failure**, I MUST append an entry to `TRIAL_LOG.md` formatted as:
```
- [YYYY-MM-DD HH:MM] vX.Y.Z : Description of attempt : Result (PASS/FAIL) : Brief notes.
```

### Persistent Learning
If a trial fails due to a hardware constraint, CubeMX configuration, or syntax issue, and I solve it, I MUST:
1. Abstract the solution into a **permanent rule**
2. Append it to `LESSONS_LEARNED.md`
3. **Read `LESSONS_LEARNED.md` before starting any new task**

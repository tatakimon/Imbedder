# RL Environment & STEVAL-STWINBX1 Project Guidelines

## 1. Workspace Architecture & Versioning (CRITICAL)
You are operating in an iterative Reinforcement Learning (RL) environment builder. 
- `Base_Firmware/`: The stable, working STM32 project. **YOU ARE STRICTLY FORBIDDEN FROM EDITING FILES IN THIS DIRECTORY.**
- `Active_Trial/`: Your working directory. All modifications happen here.
- `SUMMARY.md`: The changelog of successful implementations.

## 2. Code Modification Boundaries (ABSOLUTE RULE)
When editing files inside `Active_Trial/`, **YOU ARE FORBIDDEN FROM MODIFYING ANY CODE OUTSIDE OF `/* USER CODE BEGIN */` AND `/* USER CODE END */` BLOCKS.**

Only modify code within these protected regions:
- `/* USER CODE BEGIN PV */` ... `/* USER CODE END PV */` (Private Variables)
- `/* USER CODE BEGIN 0 */` ... `/* USER CODE END 0 */`
- `/* USER CODE BEGIN 1 */` ... `/* USER CODE END 1 */`
- `/* USER CODE BEGIN 2 */` ... `/* USER CODE END 2 */` (Setup after initialization)
- `/* USER CODE BEGIN 3 */` ... `/* USER CODE END 3 */` (Main while loop)
- `/* USER CODE BEGIN 4 */` ... `/* USER CODE END 4 */`

All HAL initialization, peripheral setup, and system configuration generated by CubeMX must remain untouched.

## 3. The RL Development Loop
Follow this strict workflow for every task:
1. **Setup**: Delete the existing `Active_Trial/` directory (if it exists) and copy the entire `Base_Firmware/` directory into a new `Active_Trial/` folder.
2. **Edit**: Make code changes strictly within `Active_Trial/` using the USER CODE boundaries.
3. **Compile**: Run `make -C Active_Trial/Debug all`. (Never run `make clean` followed by `make` unless requested).
4. **Fix**: If build fails, analyze output, correct code within boundaries, and recompile.
5. **Test**: Flash the board and monitor the serial output at `/dev/ttyACM0`. Wait for user confirmation.
6. **Promote**: ONLY when the user says "Success, promote it", copy the modified `Active_Trial/` files back into `Base_Firmware/` (overwriting the old base), and write a summary of the feature to `SUMMARY.md`.

## 4. Hardware Constraints & Timing
- **Board**: STEVAL-STWINBX1 (STM32U585AI microcontroller)
- **NEVER use `HAL_Delay()`**. Use non-blocking `HAL_GetTick()` delta comparisons:
  ```c
  if ((HAL_GetTick() - last_tick) >= desired_interval_ms) {
      last_tick = HAL_GetTick();
      // action
  }
  ```

## 7. Versioning, Logging & Memory (CRITICAL)

### Semantic Versioning
- **Base_Firmware** is always **v1.0.0**
- **Active_Trial** increments the patch version for every new compilation (e.g., v1.0.1, v1.0.2, v1.0.3...)

### Mandatory Logging
Every time you compile and test code, **regardless of success or failure**, you MUST append an entry to `TRIAL_LOG.md` formatted as:
```
- [YYYY-MM-DD HH:MM] vX.Y.Z : Description of attempt : Result (PASS/FAIL) : Brief notes.
```

### Persistent Learning
If a trial fails due to a hardware constraint, CubeMX configuration, or syntax issue, and you solve it, you MUST:
1. Abstract the solution into a **permanent rule**
2. Append it to `LESSONS_LEARNED.md`
3. **Read `LESSONS_LEARNED.md` before starting any new task**
